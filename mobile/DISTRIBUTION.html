0. The app does not run without the backend (read this first)

Building an APK is only half of what makes Swerte3 work for testers.

Piece What it is What you must do
Mobile (APK) The screen in your friend’s hand Build with EAS; set EXPO_PUBLIC_API_URL on Expo so the APK knows where to send requests.
Backend (API) Login, database, PayMongo, everything “smart” Run it (e.g. uvicorn on your PC) and expose it on the internet (e.g. ngrok) or deploy it (e.g. Google Cloud). Load backend/.env (or cloud secrets) on that server — same as when you test.
Expo environment variables do not start your API. They only tell the mobile app which public URL to call. If the backend is off, or ngrok is off, or the URL is wrong, the APK will open but sign-in / payments / data will fail — nothing is answering at that address.

For friend testing you always do both: (1) Backend running + reachable at the URL in EXPO_PUBLIC_API_URL, with backend/.env configured there, and (2) APK built with that same URL baked in (Expo preview env + eas build --profile preview).

1. What you are building

ArtifactTypical use
APK (preview profile) Email/Drive/Telegram to friends; install with “unknown sources”.
AAB (production profile) Upload to Google Play (not for direct side-load).
Expo Go Fast dev on your phone; not what you send to friends for a stable test.

The APK does not include your backend and does not include backend/.env. It only ships the React Native bundle and native shell. Your API runs separately (your PC + ngrok, or Google Cloud, etc.).

2. How the app reaches your API (important)

  • The mobile app reads EXPO_PUBLIC_API_URL (see mobile/.env.example). That value is inlined at build time when EAS runs Metro.
  • After install, the app always calls whatever URL was baked into that build. It does not read your laptop’s mobile/.env on each launch.
  • PayMongo / PayPal / DB / JWT secrets belong in backend/.env (or cloud secrets on the server). They are server-side only and are not inside the APK.

Implication for friends: Their phones must reach EXPO_PUBLIC_API_URL on the internet. http://127.0.0.1:8000, http://10.0.2.2:8000, or a random LAN IP usually will not work for someone outside your Wi‑Fi.

Ngrok: If the APK was built with https://OLD.ngrok-free.dev and you restart ngrok, the old APK still calls OLD until you rebuild (unless you use a stable ngrok domain or a fixed deployed API URL).

3. What “your server” means

Your server = whatever process answers HTTP at the base URL you put in EXPO_PUBLIC_API_URL.

Examples:

  • Local + ngrok: FastAPI on your PC, uvicorn on port 8000, ngrok forwarding public URL → that port. Secrets in backend/.env on that machine.
  • Google Cloud Run: FastAPI container in GCP. Secrets in Cloud Run env vars or Secret Manager.

The friend’s APK only does: HTTPS request → that URL. It never loads your backend/.env file.

4. Prerequisites

  • Expo account (free tier can run builds; limits apply).
  • Node.js and npm on your PC.
  • For Android APK, you do not need Xcode or a Mac.
  • Run commands from inside mobile/ where noted below.

5. Install EAS CLI and log in

npm install -g eas-cli
eas login

Directory: These two commands do not need to be run from mobile/ — -g installs globally and eas login is account-wide. Use the same Expo account you link to this project.

6. Link the project (once per app)

From mobile/ (where app.json lives):

cd path/to/Swerte3/mobile
eas init

Follow prompts to create or select the Expo project on expo.dev.

7. Configure EAS Build (once)

eas build:configure

This repo includes eas.json with:

  • preview — Android buildType: "apk" (shareable file).
  • production — Android app-bundle (Play Store).
  • development — dev client (optional).

8. Set the API URL for the build (critical)

EAS builds run in the cloud. They do not automatically see your local mobile/.env (often gitignored) unless you wire env in.

Options:

  1. EAS environment variables (recommended)
    expo.dev → Project → Environment variables → add EXPO_PUBLIC_API_URL = your public API base. No trailing slash.
    Set Environments to preview for eas build --profile preview. Variables scoped only to development are not applied to preview builds — include preview.
  2. eas.json env on the profile (commits URL to git if you put the real URL there): 
    "preview": {
  "env": {
    "EXPO_PUBLIC_API_URL": "https://your-public-api.example.com"
  },
  ...
}
  3. Secrets — See Expo EAS Environment variables.

Rule: Whatever you set for EXPO_PUBLIC_API_URL at build time is what every installed APK uses until you ship a new build.

9. Build an Android APK (friend testing)

From mobile/:

eas build --platform android --profile preview
  • First Android build: let EAS manage credentials unless you already have a keystore.
  • When the build finishes, the CLI prints a link and may show a QR code in the terminal.
  • On the web: expo.dev → your project → Builds → open the finished build (URL shape: expo.dev/accounts/…/projects/…/builds/<id>).

10. Share with testers (link, QR, or APK file)

Option A — Send the link (and QR from the dashboard)

  • Just the URL: Copy the build page URL from the browser (or from the CLI) and send it. Friends open it on Android (Chrome) and use Install.
  • QR on expo.dev: On that page, find Build artifact (with the APK label). The QR is not on the main screen — click the white Install button. Expo then shows a step (modal / panel) with the QR code to scan from an Android phone.
  • Terminal: After eas build completes, a QR may already appear in the terminal for scanning.
  • Some internal / preview flows may ask testers to sign in with a free Expo account. To skip that, use Option B.

Option B — Download the .apk and share the file

  1. On expo.dev, open the same finished build.
  2. In Build artifact, open the ⋮ (three-dot) menu next to Install / Open with Orbit and choose Download (wording may vary), or use Artifacts if shown.
  3. Save the .apk, then share via Drive, Telegram, email (within size limits), etc.
  4. Friends download on the phone, open the file, and install (§11).

They must trust you — these installs are not from the Play Store.

11. Tester checklist (Android)

  1. Settings → Apps → Special access (or Security) → allow Install unknown apps for Chrome / Files / Drive.
  2. Open the downloaded .apk → Install.
  3. If Play Protect warns, confirm only if they trust the source.

12. Common failures

SymptomLikely cause
Login / network errors for friends EXPO_PUBLIC_API_URL not public, wrong, or PC/ngrok off.
Worked yesterday, broken today (ngrok) Ngrok URL changed; rebuild APK or use stable/deployed URL.
Build OK but wrong API Env not set for EAS preview; fix env and rebuild.
“App not installed” Corrupt download or ABI mismatch — re-download or rebuild.
eas build upload fails — git clone … code: 128 (Windows) Often dubious ownership on .git. Run once (adjust path): git config --global --add safe.directory "C:/Users/jonel/OneDrive/Desktop/Jonel_Projects/Swerte3/.git" then eas build again.
EPERM: scandir '…\.pytest_cache' Root .easignore skips backend/ on EAS upload. Commit it before eas build.

13. Production / Play Store (later)

eas build --platform android --profile production

You get an AAB. Upload to Google Play Console. Keep EAS-managed signing for updates.

14. iPhone / iOS — sharing with friends (detailed)

iOS is stricter than Android: there is no casual “send a file like an APK” for a standalone app. For that, testers use Apple’s system — almost always TestFlight — which needs the paid Apple program (§14.2).

14.0 I only want to test on my iPhone — pay $99?

No if you use Expo Go:

  1. Install Expo Go on the iPhone (free).
  2. From mobile/, run npx expo start.
  3. Same Wi‑Fi or tunnel, then scan the QR from the terminal / dev tools.

No Apple Developer Program fee for that. The app runs inside Expo Go, not your final standalone build — custom native modules may force a dev build or the paid path.

~$99/year applies when you want a real EAS iOS build → TestFlight / App Store.

14.1 What you need for TestFlight / standalone iOS builds (once)

Requirement Why
Apple Developer Program (~$99/year) For TestFlight, App Store, typical EAS iOS standalone builds — not for Expo Go-only testing (§14.0).
Apple ID enrolled in that program Signing and App Store Connect.
EXPO_PUBLIC_API_URL on EAS Same as Android (§8) — set for preview or production for the iOS build.
ios.bundleIdentifier in app.json Unique app ID (e.g. com.jonel0322.swerte3). Keep it stable for the App Store.

You do not need your own Mac for compiling if you use EAS Build (cloud).

14.2 Typical path: EAS → App Store Connect → TestFlight

  1. Register the bundle ID if needed: Apple Developer → Identifiers — App ID matching app.json → ios.bundleIdentifier.
  2. App Store Connect: appstoreconnect.apple.com → Apps → + → New App (iOS, same bundle ID). Minimal metadata is enough to use TestFlight.
  3. First iOS build from mobile/: 
    eas build --platform ios --profile preview
    Follow EAS prompts (Apple ID, App Store Connect API key, or app-specific password) — see Expo app credentials.
  4. Wait for the build on expo.dev → your project → Builds. iOS produces an IPA; you usually do not email this to friends.
  5. Submit to TestFlight if not automatic: 
    eas submit --platform ios --profile production
    Or upload via Transporter / Expo — EAS Submit (iOS). Build must appear under your app → TestFlight in App Store Connect.
  6. Add testers in App Store Connect → TestFlight: Internal (up to 100, users in Users and Access) vs External (email list; first external group often needs Beta App Review).
  7. Friends: Install TestFlight from the App Store → open Apple’s invite email or a public link → Accept → Install Swerte3.

14.3 Android vs iPhone

Android (§10) iPhone
Link or .apk TestFlight invite or public link — not a casual .ipa mail
Often no Apple account Free Apple ID needed for TestFlight
“Install unknown apps” Only TestFlight / App Store

14.4 Backend

Same as Android: your API (ngrok or deployed URL) must be up at EXPO_PUBLIC_API_URL.

14.5 Other options

Ad hoc / UDIDs — possible for small lists, heavy setup. Expo Go — fine for you in dev, not a stand-in for TestFlight with external testers.

Docs: Expo iOS build, TestFlight.

15. One-line recap

Android: Set EXPO_PUBLIC_API_URL for preview → eas build --platform android --profile preview → share link / QR / .apk (§10) → keep API running.

iPhone: Apple Developer Program → eas build --platform ios --profile preview (and submit to TestFlight) → friends use TestFlight (§14) → same API URL env as Android.

For payments/webhooks, see backend/.env.example and PayMongo (webhook URL must match your public API host).
